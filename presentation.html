‚ö° The Problem

  • Does more training data = better performance?
  • Prior studies focus on performance metrics only
  • Energy costs are underrepresented in research
  • Need to integrate power consumption into analysis
  • Question: Are token increases energetically efficient?

üî¨ Experimental Design

  • Model: 1.1B parameter TinyLlama
  • Three token counts: 500K, 1M, 2M
  • Repeated-measures design on constant GPU
  • Fixed architecture, optimizer, epoch counts
  • Tracked: performance + power + execution time

üìè Efficiency Metrics

Training efficiency is defined as performance gain per unit energy and wall-clock time.

We normalize by the 1M-token run to compare across runs on identical hardware.

  • Performance: accuracy and loss trends
  • Energy: total GPU power draw (Wh)
  • Time: wall-clock runtime per epoch
  • Parameter efficiency: performance per parameter per token

üìä Key Findings

  • Performance metrics: inconsistent/diminishing returns
  • Training efficiency: strictly monotonic decline
  • Strong effect of token count on parameter efficiency
  • All pairwise comparisons significant (Bonferroni corrected)
  • More tokens ‚â† proportionally better results

üí° Implications

  • Training token increases may be energetically inefficient
  • True even when marginal performance improves
  • Need efficiency-aware evaluation in LLM training
  • Power consumption must be a first-class metric

üîç Statistical Significance ‚â† Performance Gain

Key insight: Bonferroni-corrected p-values show all power consumption differences are highly significant (p < 0.001), yet performance metrics are inconsistent or diminishing.

  • Strict monotonic increase in power and cost
  • Inconsistent gains in inverse perplexity
  • Statistical significance does not imply practical utility
  • Energy efficiency must be weighed against marginal improvements
  • The tradeoff space is much more complex than typical scaling laws suggest

üß≠ Limitations & Next Steps

  • Single model size and hardware configuration
  • Token counts limited to three settings
  • Future work: broader model scales + varied optimizers
  • Integrate carbon intensity and cost per token
  • Explore efficiency-aware early stopping
